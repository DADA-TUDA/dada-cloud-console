# ADR-006: v1 Complete — v2 Scope Opened

## Status

Accepted — 2026-05-08

## Context

DADA Cloud Console v1 has been deployed to the beget-prod cluster.
The core product hypothesis is validated:

```
UI click → Operation record → Worker renders YAML → Git commit → Argo sync → ServiceDatabase created
```

Evidence:
- Generated `ServiceDatabase` manifest confirmed in state repo under `clusters/beget-prod/projects/client-a/environments/prod/apps/profi-backend/database.yaml`
- Jenkins CI pipeline builds and deploys backend + frontend artifacts
- Helm chart deployed to cluster
- Operation timeline and audit log functional
- Role-based views (client / developer / admin) operational

ADR-001 through ADR-005 are all in `Accepted` status.

## Decision

v1 is declared complete as of 2026-05-08.

v2 scope is opened as defined in the ROADMAP, focused on **managed service expansion**:

| Feature | Priority | Complexity |
|---------|----------|------------|
| v2.1 App lifecycle (image update, scale, restart, rollback) | P0 | Medium — extends existing operation model |
| v2.2 Advanced gateway/domain management | P1 | Medium — new typed action + frontend |
| v2.3 Restore requests | P1 | High — admin approval flow required |
| v2.4 Redis managed service | P2 | Low — pattern established by ServiceDatabase |
| v2.5 RabbitMQ managed service | P2 | Low — same pattern |
| v2.6 Secret rotation | P2 | Medium — sensitive ops, approval required |

### v2 prerequisite (from v1 gaps)

Before v2.1 is complete, the App resource must be generated by the worker.
Currently `ServiceDatabase` references `appRef`, but no `App` manifest exists in state repo.

This is treated as part of v2.1, not a v1 blocker.

## v2 Exit Criteria

- User can update image tag of a running app from UI
- User can scale replicas without writing YAML
- User can add a second domain to an app
- User can create a restore request
- Redis is available as a managed add-on
- All new actions follow the Operation model (ADR-003)
- All new resources are typed actions (ADR-004)

## Consequences

- v2 work begins immediately
- App manifest generation (worker) is the first v2 backend task
- Frontend image-update and scale UI can be built in parallel
- The operation model from ADR-003 requires no changes — v2 adds new action types only
- v2.3 restore and v2.6 rotation require the WaitingForApproval state from ADR-002 to be wired in the UI (not implemented in v1)

## Supersedes

None. Extends ADR-001 scope.
